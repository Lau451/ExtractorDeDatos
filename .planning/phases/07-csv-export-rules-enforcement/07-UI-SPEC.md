## Color

All colors are defined as CSS custom properties in `src/index.css` using OKLCH.
The neutral base-nova preset has zero chroma (achromatic) — accent behavior is expressed through the primary token.

| Role | CSS Token | Light Value | Usage |
|------|-----------|-------------|-------|
| Dominant (60%) | `--background` | oklch(1 0 0) — white | Page background, `min-h-screen bg-background` |
| Secondary (30%) | `--card` / `--muted` | oklch(1 0 0) / oklch(0.97 0 0) — near-white | Cards (`bg-card`), muted surfaces, table backgrounds |
| Accent (10%) | `--primary` | oklch(0.205 0 0) — near-black | Reserved-for list below |
| Destructive | `--destructive` | oklch(0.577 0.245 27.325) — red | Error icon, error button border, error text |
| Warning (new, Phase 7) | amber/yellow utility class | `bg-amber-50 border-amber-200 text-amber-800` | Export warning banner only — uses Tailwind utility, not a CSS custom property |

**Accent reserved for:**
- Primary action buttons (`variant="default"` — "Download CSV")
- Focus ring outlines
- Selected/active state indicators in DocTypeBar select

**Warning color rationale:** The `--accent` token in the neutral preset maps to the same muted surface as `--secondary` (oklch 0.97) — it carries no semantic warning meaning. The export warning banner must use Tailwind amber utilities directly (`bg-amber-50 border-amber-200 text-amber-800`) to signal a non-blocking caution state, distinct from the destructive red used for errors.

Source: src/index.css CSS custom properties + App.tsx + ProgressView.tsx usage patterns.

## New Interaction: Export Warning Banner

### Trigger
After `handleDownloadCSV` completes its `fetch()` call and `res.headers.get('X-Export-Warnings')` returns a non-null value.

### Appearance
```
┌─────────────────────────────────────────────────────┐
│  ⚠  Some fields could not be verified               │
│     Missing: reference_number, issue_date           │
│     The CSV was downloaded. Review these fields     │
│     before using the file.                          │
└─────────────────────────────────────────────────────┘
```

- Container: `rounded-lg border border-amber-200 bg-amber-50 p-4 mt-4`
- Icon: `AlertTriangle` from lucide-react, `h-4 w-4 text-amber-600 mr-2`
- Heading: `text-sm font-semibold text-amber-800` — "Some fields could not be verified"
- Body: `text-sm text-amber-700` — "Missing: {comma-separated field list}"
- Sub-body: `text-sm text-amber-700` — "The CSV was downloaded. Review these fields before using the file."
- Layout: heading + icon on one line; body text below

### State management
- Add `exportWarnings: string[] | null` to App component state, initialized to `null`
- Set to `warnings.split(',').map(w => w.trim())` when `X-Export-Warnings` header is present
- Clear to `null` on `handleReset()` and on new download attempt (before fetch)
- The banner renders whenever `exportWarnings` is non-null and non-empty, in both `review` and `done` phases

### No banner when
- `X-Export-Warnings` header is absent (all mandatory fields present)
- State is `upload` or `processing`

### No blocking behavior
The download proceeds regardless. The banner is informational only — no "confirm before download" modal, no disabled state on the button.

Source: CONTEXT.md decisions — "Warn but allow download. Export returns HTTP 200."

## Copywriting Contract

| Element | Copy | Source |
|---------|------|--------|
| Primary CTA | "Download CSV" | Phase 5 App.tsx — unchanged |
| Warning banner heading | "Some fields could not be verified" | Default (no upstream spec) |
| Warning banner field list | "Missing: {field_1}, {field_2}" | Derived from X-Export-Warnings header value |
| Warning banner action guidance | "The CSV was downloaded. Review these fields before using the file." | Default |
| Empty state heading | "Drop your file here" | Phase 5 UploadZone — unchanged |
| Empty state body | "or click to browse — PDF, XLSX, PNG, JPG, HTML" | Phase 5 UploadZone — unchanged |
| Error state heading | "Processing failed" | Phase 5 ProgressView — unchanged |
| Error state body | Delegated to `getErrorMessage(errorCode)` utility | Phase 5 — unchanged |
| Post-download reset CTA | "Upload another document" | Phase 5 App.tsx — unchanged |
| Retry CTA | "Try again" | Phase 5 ProgressView — unchanged |

**Field name display in warning:** Use raw snake_case field names from the `X-Export-Warnings` header value directly (e.g., `reference_number`, `issue_date`). Do not map to human-readable labels — the warning targets the user who will PATCH corrections, not a first-time viewer.

**No destructive actions in this phase.** No confirmation dialogs needed.

## Interaction States

### "Download CSV" button — review phase
| State | Appearance |
|-------|-----------|
| Default | `variant="default"` — near-black background, white text, Download icon |
| Hover | shadcn default hover (slightly lighter primary) |
| Active / pressed | shadcn default active |
| Loading (during fetch) | Button disabled, spinner replaces Download icon, text "Downloading..." |
| Post-download (warning) | Button re-enabled; warning banner appears below |
| Post-download (no warning) | Phase transitions to `done` immediately |

**Loading state detail:** Set `isDownloading: boolean` state. While `isDownloading === true`, render `<Loader2 className="animate-spin mr-2 h-4 w-4" />` instead of `<Download />` and set `disabled` on the Button. Re-enable after `fetch()` resolves (whether or not a warning is present).

Source: RESEARCH.md Pattern 6 — fetch is async; CONTEXT.md — "warn but allow download."

## Accessibility

- Warning banner: `role="alert"` on the container div so screen readers announce it after download
- Amber color contrast: `text-amber-800` on `bg-amber-50` — WCAG AA compliant (contrast ratio ~7:1)
- Loading button: `aria-disabled="true"` while `isDownloading === true`
- Field names in warning are machine-readable (snake_case) — acceptable; the warning targets analysts who know the field vocabulary
