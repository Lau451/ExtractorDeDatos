## Phase Nature

Phase 4 is a **backend-only phase**. There is no rendered UI in this phase. The "interface" is the REST API surface — the HTTP request/response shapes, error vocabulary, and job state machine that Phase 5's React SPA will consume.

This document defines the **API interaction contract** in place of a visual design contract. All spacing, typography, and color tokens are marked `not applicable` for this phase and will be defined in the Phase 5 UI-SPEC. The sections that matter here are: copywriting (error messages, state labels), state machine, and response shapes.

## API Interaction Contract

This section is the authoritative source of truth for Phase 4. Phase 5 executor reads this before building any UI that calls these endpoints.

### Endpoint Inventory

| Method | Path | Purpose | Success Code |
|--------|------|---------|--------------|
| PATCH | /jobs/{id}/fields | Apply user corrections to extraction_result | 200 |
| GET | /jobs/{id} | Poll job status (existing, unchanged) | 200 |
| GET | /jobs/{id}/export | Download CSV with corrections applied (existing, unchanged) | 200 |

### PATCH /jobs/{id}/fields — Request Shape

```json
{
  "fields": {
    "field_name": "new_value"
  }
}
```

- `fields` is a flat or nested dict. Keys not present in the request retain their current value.
- List elements (e.g. `line_items`) are merged by index — only the specified index fields are updated.
- Source: CONTEXT.md locked decision

### PATCH /jobs/{id}/fields — Response Shape (200)

Returns the full updated `JobResponse`. Client uses this to confirm applied edits without a second GET call.

```json
{
  "job_id": "string",
  "status": "complete",
  "doc_type": "string",
  "extraction_result": { "...updated fields..." },
  "error_code": null,
  "error_message": null
}
```

- Source: CONTEXT.md locked decision + RESEARCH.md models.py

### Job State Machine

```
pending → processing → complete
                     ↘ error
```

| State | Meaning | Patchable |
|-------|---------|-----------|
| pending | Job created, not yet processing | No (409) |
| processing | Docling parsing or Gemini extraction in progress | No (409) |
| complete | Extraction finished, result available | Yes (200) |
| error | Pipeline failed — see error_code | No (409) |

Only `complete` jobs accept PATCH. All other states return 409.

### Error Code Vocabulary

All five error codes are defined in `src/core/errors.py`. Phase 5 UI maps each code to a human-readable message.

| Code | Trigger | Human-Readable Label (Phase 5 renders this) |
|------|---------|----------------------------------------------|
| `DOCLING_TIMEOUT` | Document parsing exceeded timeout | "Document took too long to parse. Try again with a smaller file." |
| `DOCLING_PARSE_ERROR` | Docling failed to parse the file | "Could not read this file. Check that it is not corrupted." |
| `GEMINI_ERROR` | LLM call failed or returned invalid response | "Extraction failed. Please try uploading the document again." |
| `INVALID_FILE_TYPE` | File extension/MIME not in supported list | "This file type is not supported. Use PDF, XLSX, PNG, JPG, or HTML." |
| `FILE_TOO_LARGE` | File exceeds size limit | "File is too large. Reduce the file size and try again." |

Source: CONTEXT.md locked decision

### TTL / Job Expiry

- Jobs expire 1 hour after creation.
- Expired jobs return 404 — identical response to unknown job ID.
- Phase 5 must treat 404 on a previously-known job ID as "session expired" and prompt the user to re-upload.
